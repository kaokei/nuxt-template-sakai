## 二、权限模型设计

### 2.1 核心设计决策：菜单即权限

权限标识（`permission` 字段）直接挂在菜单节点上，角色通过 `menuIds` 关联菜单项来获得权限。

```
User ──(roleIds)──▶ Role ──(menuIds)──▶ Menu
                                           │
                    Menu.type = 'button' ──┘ (按钮级权限)
                    Menu.permission = "system:user:add" (权限标识)
```

**Role 模型**：`id / name / code / dataScope / menuIds[] / deptIds[] / status`

**Menu 模型**：`id / name / parentId / type(directory|menu|button) / permission / route / icon / visible / status`

已有权限标识举例：

| 菜单项   | 类型   | permission           |
| -------- | ------ | -------------------- |
| 用户管理 | menu   | `system:user:list`   |
| 用户新增 | button | `system:user:add`    |
| 用户编辑 | button | `system:user:edit`   |
| 用户删除 | button | `system:user:delete` |
| 角色管理 | menu   | `system:role:list`   |

### 2.2 为什么不在菜单之上再抽象一层独立的权限表？

备选方案是引入独立的 Permission 实体，Menu 通过 permissionId 引用它。这带来了额外的抽象层和复杂度，而收益有限。

**选择菜单即权限的理由：**

1. **简单直接** — 管理员在菜单树中就能完成"谁能访问哪个页面、谁能点哪个按钮"的配置，不需要在两个页面间跳转
2. **若依（Ruoyi）验证过的模式** — 国内最广泛使用的 RBAC 参考实现，生产环境验证
3. **按钮级权限天然支持** — `type=button` 的菜单节点本身就是权限载体

### 2.3 权限码重复的权衡

当一个功能（如"导出"）被多个页面共用时，需要在各自的菜单节点下配置相同的 permission 字符串。这带来了一定的数据冗余，但**可以接受**，因为：

- 实际上不同页面的"导出"导出的是不同数据，"用户管理导出"和"角色管理导出"在业务上本身就是两个不同的权限
- 如果确实需要"全局导出"语义，配置相同的 permission 字符串（如 `system:export`）即可，`hasPermission("system:export")` 对两者同时生效
- 相比引入独立权限表带来的模型复杂度和维护成本，这点冗余的代价更小

### 2.4 数据权限为什么挂在角色上

当前设计中，`Role.dataScope` 直接挂在角色上，而非菜单节点上。这与功能权限（`Menu.permission`）的载体不同。

**功能权限 vs 数据权限 — 本质区别：**

```
功能权限：能不能干？
  例：你能否点击"删除用户"按钮？
  载体：Menu.permission = "system:user:delete"
  粒度：每个操作一个开关

数据权限：能看谁的数据？
  例：你能看到全公司用户，还是只能看自己部门的？
  载体：Role.dataScope = all | deptAndBelow | dept | self | custom
  粒度：人的组织边界，跨页面统一
```

**为什么挂在角色上是合理的：**

数据权限描述的是"**这个人能看多大范围**"，不是"**在这个页面能看多大范围**"。一个部门经理无论在用户管理、角色管理还是日志页面，他都只应该看到自己部门及下属部门的数据。这是人的组织属性，不应该随访问的页面而变化。

| 角色       | dataScope      | 含义         |
| ---------- | -------------- | ------------ |
| 超级管理员 | `all`          | 全公司       |
| 部门经理   | `deptAndBelow` | 本部门及下属 |
| 普通员工   | `self`         | 仅本人       |

若依（Ruoyi）同样采用此设计：功能权限在菜单，数据权限在角色，职责清晰。

**是否存在例外？**

如果出现"用户 A 在页面 X 看全部数据，在页面 Y 只看自己数据"的极端需求，通常通过给用户分配两个角色来解决，而不是把数据权限下沉到菜单级。改动模型来迁就极端场景是过度设计。

> **结论**：当前设计合理，不需要改变。功能权限（操作级）在菜单节点上，数据权限（组织边界）在角色上，二者职责分明。

## 三、缺失分析

当前系统只完成了 RBAC 的**数据模型和管理后台**。整个**运行时权限执行层**完全缺失：

| 缺失能力       | 现状                                                                    |
| -------------- | ----------------------------------------------------------------------- |
| 登录/认证      | `user.service.ts` 硬编码 `userId:9527, username:'游客'`，无真实认证流程 |
| 动态菜单过滤   | 侧边栏菜单是静态配置，所有角色看到完全相同的菜单                        |
| 按钮级权限控制 | 页面上的"新增""编辑""删除"按钮无条件渲染，没有 `v-permission` 指令      |
| 路由守卫       | 无 `middleware/` 目录，任何人都能直接访问任何页面 URL                   |
| 权限校验函数   | 无 `usePermission()` / `hasPermission()` 等 composable                  |

**一句话总结**：数据层完整（谁有什么权限已经能配置），但执行层空白（配置好的权限没有被任何代码消费）。
