## base

不能使用useService+declareProviders

受控组件+无状态组件，也就是完全和业务无关的组件，多个项目可以复用的组件

- 无状态展示组件
- 受控组件
- 基础UI组件
- 业务无关组件，任何页面都能复用的组件，不是特定页面复用的组件

## common

可以使用useService+declareProviders
只不过这里declareProviders主要用于声明组件自身的服务。

多个页面共享的业务组件，主要用于解决A页面和B页面需要共享的组件。
一般是存放在A页面和B页面的公共父级页面，或者直接放在features目录下面。
又或者一开始是A页面和B页面共用，后来又有C页面和D页面共用，放在哪里都不合适，所以单独设立一个common文件夹。

## features

可以使用useService。
尽量不要在这里使用declareProviders，如果使用了declareProviders，那么说明这个组件是期望可复用的组件，应该移动到common文件夹中。

页面入口组件，和页面强相关的业务组件，目录结构基本和pages一致，是实现业务逻辑的主要文件夹。

把原本的pages目录中的业务逻辑抽象成一个页面组件以方便在不同页面中复用。

- 业务组件，和页面路由强绑定的组件
- 页面入口组件，一般是非受控组件
- 页面容器组件，一般是非受控组件
- containers 容器组件+功能组件【弹窗组件】

## pages

页面入口，不包含业务逻辑，所有业务逻辑都由features中的组件实现。

页面级别的服务应该在pages目录中进行声明，
如果是嵌套页面，也可以继续在二级路由中继续声明二级页面的服务。

## 总结

一般来说，应该在pages目录中声明各个页面级别的服务。

中间的容器组件则尽量不需要声明自己的服务，而是从页面服务中获取数据。

功能组件也可以有自己的服务和数据，页面组件则可以通过命令式方法查找对应的功能组件中的服务。
也就是和useService对应的findService方法。
